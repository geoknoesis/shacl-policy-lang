Abstract

SHACL Policy Language (SPL) v0.3 defines a profile of W3C SHACL 1.2 for attribute-based access control over RDF access-request graphs, including Verifiable Credentials as evidence. A Policy Decision Point evaluates SPL policies and returns a normative Decision graph (shpl:Decision) capturing shpl:outcome, reasons, optional obligations, and a request reference. Deployment roles align with [[XACML30]]; attribute categories align with [[NIST-SP-800-162]].

All rule conditions are expressed as SHACL sh:NodeShape values of shpl:condition (no parallel constraint language). SPL-Core is limited to [[SHACL12-CORE]], [[SHACL12-RULES]], and [[SHACL12-NODE]]; SPL-Full additionally allows [[SHACL12-SPARQL]]. SPL defines a compact shpl: vocabulary for effects, targets, actions, duties, obligations, conflict combination, hooks, and capability discovery; action IRIs SHOULD reuse [[ODRL]] where practical. Transport, signing, negotiation, and policy merge are out of scope for core and are deferred to Companion profiles.

Goals and background

Problems addressed

Organizations increasingly govern access with semantic data (RDF graphs, catalogs, knowledge bases) and Verifiable Credentials as evidence. That raises a set of practical problems SPL is designed to solve:

  • Same stack for policy and data. When policies and resources live in RDF, ad hoc policy formats force extra bridges, parsers, and duplicated attribute models. SPL keeps policy, requests, and decisions as RDF so one graph stack can be stored, queried, versioned, and signed consistently.
  • One constraint language. Credential and request checks (age, role, purpose, graph shape) are fundamentally validation problems. Inventing a second constraint dialect for policy duplicates tooling and skills. SPL uses SHACL 1.2 only under shpl:condition, so the same SHACL engines and test patterns apply to policy as to data [[SHACL12-CORE]].
  • Portable decisions. Enforcement, audit, and federation need a stable, machine-readable authorization decision (permit or deny, why, and what obligations follow). SPL standardizes a shpl:Decision graph aligned with common PDP/PEP separation [[XACML30]].
  • Interop with existing policy concepts. ODRL already defines permissions, prohibitions, duties, assets, and actions [[ODRL]]. SPL maps those ideas onto SHACL-evaluated rules without replacing ODRL negotiation or legal prose. Attribute categories follow ABAC practice [[NIST-SP-800-162]].

Design rationale

SPL is intentionally a thin profile: a compact shpl: vocabulary for what SHACL does not name by itself (effects, target/action applicability, combining multiple rules, duties, obligation payloads, negative outcomes, hooks, and processor capabilities). Everything expressible as a condition on the request stays in SHACL. Wire formats, offer/agreement negotiation, and merge semantics stay in companion profiles so the core stays testable and implementable in one place.

SPL in five minutes

The following example illustrates the ideas in Goals and background. An SPL policy is an RDF resource typed shpl:Policy that carries one or more shpl:Rule resources.

Each rule has exactly one shpl:effect (shpl:Permit, shpl:Deny, or shpl:Obligate), at least one shpl:target, exactly one shpl:action, and optionally a shpl:condition — a sh:NodeShape evaluated with the access request as focus node. If shpl:condition is absent, the condition is satisfied whenever target and action match.

@prefix shpl: <https://w3id.org/shacl-policy-language#> .
@prefix odrl: <http://www.w3.org/ns/odrl/2/> .
@prefix sh:   <http://www.w3.org/ns/shacl#> .
@prefix cred: <https://www.w3.org/2018/credentials#> .
@prefix ex:   <http://example.com/ns#> .

<> a shpl:Policy ;
   shpl:profile shpl:Core ;
   shpl:specVersion <https://w3id.org/shacl-policy-language/spec/v0.3.0> ;
   shpl:conflictStrategy shpl:DenyOverrides .

ex:AdultRead a shpl:Rule ;
   shpl:effect shpl:Permit ;
   shpl:target ex:AdultOnly ;
   shpl:action odrl:read ;
   shpl:condition [
     a sh:NodeShape ;
     sh:property [
       sh:path shpl:credential ;
       sh:qualifiedValueShape [
         a sh:NodeShape ;
         sh:property [
           sh:path cred:credentialSubject ;
           sh:node [ sh:property [ sh:path ex:age ; sh:minInclusive 18 ] ]
         ]
       ] ;
       sh:qualifiedMinCount 1
     ]
   ] ;
   shpl:duty [
     a shpl:Duty ;
     shpl:triggersOn shpl:OnRuleApplied ;
     shpl:timing shpl:Postcondition ;
     sh:rule [
       a sh:TripleRule ;
       sh:subject sh:this ;
       sh:predicate ex:auditSink ;
       sh:object <https://example.com/audit>
     ]
   ] .

ex:Req1 a shpl:AccessRequest ;
  shpl:agent <did:example:alice> ;
  shpl:action odrl:read ;
  shpl:target ex:AdultOnly ;
  shpl:credential ex:AliceAgeVC .

ex:Dec1 a shpl:Decision ;
  shpl:outcome shpl:Permit ;
  shpl:appliedRule ex:AdultRead ;
  shpl:obligation [
    a shpl:Obligation ;
    shpl:fromRule ex:AdultRead ;
    shpl:timing shpl:Postcondition ;
    shpl:state shpl:ObligationPending ;
    ex:auditSink <https://example.com/audit>
  ] ;
  shpl:requestRef ex:Req1 .

What SPL adds on top of raw SHACL: rule effects, applicability wiring (target + action + optional action closure), policy-level conflict strategies, duty attachment through shpl:duty, obligations materialized through SHACL rules, a split negative outcome model (shpl:NotMatched vs shpl:NotSatisfied), and a normative Decision graph. That division of labor is summarized under Design rationale.

Relationship to SHACL 1.2

SPL is not a separate constraint language: that is the main rationale for a SHACL-only condition model (see Goals and background). Everything under shpl:condition is evaluated by a SHACL 1.2 engine. Contextual values (time, counters, purpose) use SHACL 1.2 node expressions per [[SHACL12-NODE]]. Named hooks such as shpl:currentDateTime are registered in the SPL vocabulary; profiles MAY add more.

Relationship to ODRL

ODRL [[ODRL]] provides the permission, prohibition, duty, asset, and action information model and the recommended action IRIs. SPL reuses those concepts for interoperability while keeping conditions in SHACL (see Goals and background). SPL maps constructs to SHACL-backed rules and requests as summarized in Appendix A and Conceptual alignment. SPL does not reimplement ODRL negotiation or legal prose; it standardizes SHACL-backed evaluation and Decision graphs.

Companion profiles (non-normative)

  • Trust registry graphs and federation wire formats
  • ODRL-style policy negotiation lifecycle, ordered rule sets, policy inheritance, and merge algorithms
  • Compact authoring syntax expanded before validation

Conceptual alignment

This section is informative. It relates SPL to the access-control reference model in [[XACML30]] and to attribute categories in [[NIST-SP-800-162]] so implementers can map SPL artifacts to familiar security architecture without changing the normative rules in Evaluation semantics and Vocabulary.

Reference architecture (XACML)

The following uses the same component names and relationships as the reference architecture for the eXtensible Access Control Markup Language [[XACML30]]:

  • Policy Administration Point (PAP) — authors, version-controls, or distributes shpl:Policy graphs.
  • Policy Decision Point (PDP) — evaluates applicable policy against an access request and produces a shpl:Decision (authorization decision plus any returned obligations in the XACML sense).
  • Policy Enforcement Point (PEP) — constructs or forwards the shpl:AccessRequest, invokes the PDP, enforces shpl:outcome (shpl:Permit / shpl:Deny and default-deny handling for shpl:NotMatched / shpl:NotSatisfied per default deny at the enforcement boundary), and fulfills obligations by advancing shpl:state on each shpl:Obligation.
  • Policy Information Point (PIP) — supplies attribute values the PDP needs when they are not already present in the request graph, in particular for environmental or external hooks.

Wire protocols for carrying requests and decisions (for example authorization frameworks built on Hypertext Transfer Protocol) are out of scope for SPL core; they belong in companion profiles and are orthogonal to this decision model, in the same way that transport is orthogonal to the XACML reference architecture [[XACML30]].

Attribute categories (ABAC)

[[NIST-SP-800-162]] groups data used in ABAC decisions into subject, resource, action, and environment attribute categories. In SPL v0.3 the Access request graph plays the role of the attribute context presented for evaluation:

ABAC category [[NIST-SP-800-162]]Primary SPL v0.3 carriers
Subjectshpl:agent and claims in shpl:credential / cred:credentialSubject validated by shpl:condition
Resourceshpl:target (the governed resource; see Policy and rules for alignment with odrl:Asset in [[ODRL]])
Actionshpl:action (IRIs from the ODRL action vocabulary [[ODRL]] are recommended)
EnvironmentDeployment-supplied triples in the Access request graph, if any, plus values bound for node expression hooks (for example shpl:currentDateTime)

All constraint logic over those attributes remains SHACL [[SHACL12-CORE]] on the Access request graph as described in Conditions.

Decisions and rule combining

[[XACML30]] defines authorization decisions including Permit, Deny, and Indeterminate, and defines rule-combining algorithms and policy-combining algorithms for aggregating multiple evaluated rules. SPL maps as follows (SPL terms are RDF individuals on shpl:Decision / per-rule classification in Evaluation semantics):

XACML decision [[XACML30]]SPL shpl:outcomeNote
Permitshpl:PermitSame access result label is reused on rules as shpl:effect where applicable.
Denyshpl:DenyIdem for shpl:effect.
Indeterminateshpl:IndeterminateUse shpl:reason for a machine-readable cause.
NotApplicable (no applicable policy)shpl:NotMatchedNo rule's target and action matched the request.
—shpl:NotSatisfiedTarget and action matched at least one rule, but no satisfied Permit/Deny rule decided access; typically a violated shpl:condition. This refines “no applicable positive decision” using SHACL's pass/fail outcome.

For combining, shpl:DenyOverrides and shpl:PermitOverrides are the deny-overrides and permit-overrides strategies described in [[XACML30]] for merging rule or policy effects. Optional shpl:detectConflicts detects a Permit/Deny conflict and yields shpl:Indeterminate before applying the strategy. Other combining algorithms in [[XACML30]] are not in SPL v0.3 core and MAY be introduced by profiles.

[[XACML30]] also distinguishes obligations (enforceable by the PEP) from advice (non-normative hints). SPL core represents obligations on a shpl:Decision as shpl:Obligation nodes linked via shpl:obligation; it does not define an advice class.

Terminology

Policy Decision Point (PDP)
As in [[XACML30]]: the entity that evaluates policy and returns an authorization decision. In SPL it evaluates rules and emits a shpl:Decision.
Policy Enforcement Point (PEP)
As in [[XACML30]]: the entity that enforces the decision. In SPL it enforces shpl:outcome and fulfills obligations (nodes of type shpl:Obligation) linked from the shpl:Decision.
Policy Information Point (PIP)
As in [[XACML30]]: the entity that provides attribute values to the PDP. In SPL it supplies values for hooks when they are not already present in the Access request graph.
Policy Administration Point (PAP)
As in [[XACML30]]: the entity that manages policies. In SPL it authors or distributes SPL policies as shpl:Policy graphs.
SPL policy
An RDF resource typed shpl:Policy that carries one or more shpl:Rule resources for evaluation.
Access request graph
The RDF graph describing one shpl:AccessRequest and its attached credential nodes, used as data for SHACL evaluation. It is the attribute context presented to the PDP in the ABAC sense [[NIST-SP-800-162]].
Decision graph
The RDF graph describing one shpl:Decision, including the authorization decision (shpl:outcome), reasons, optional obligations, and a reference back to the request [[XACML30]].
Rule condition
The optional sh:NodeShape value of shpl:condition on a rule; evaluated against an Access request graph.
focus node
Per [[SHACL12-CORE]], the RDF term validated by a shape. For each rule's shpl:condition, the focus node is the shpl:AccessRequest being decided (see SHACL focus node for conditions).
Execution profile
shpl:Core or shpl:Full; declares which SHACL 1.2 facilities are in play.
Domain profile
A shpl:Profile referenced by shpl:conformsToProfile that declares extra capabilities, shapes, or vocabularies beyond core SPL.
Capability
An IRI naming an optional language or processor feature that SPL policies may require and PDPs may advertise.
Duty
In [[ODRL]], a duty is an obligation on a permission or party. In SPL, the same idea is a shpl:Duty attached with shpl:duty, with a trigger and SHACL rules that emit payload triples (see Appendix A).

Conformance

The key words MAY, MUST, MUST NOT, OPTIONAL, RECOMMENDED, REQUIRED, SHALL, and SHOULD are to be interpreted as described in BCP 14 [[RFC2119]] [[RFC8174]] when, and only when, they appear in all capitals in this specification.

Conformance is expressed in terms of SPL policy graphs (see Conformance classes) that validate to SPL-SPL, execution profiles (SPL-Core versus SPL-Full), and processors (PDPs that implement evaluation and Decision graph output as specified below).

Conformance classes

The following conformance classes are normative. A product MAY claim conformance to one or more classes independently.

ClassDefinitionNormative requirements
SPL policy graph (document)An RDF graph (or graph set) whose root policy resource is typed shpl:Policy and that is intended for interchange or validation.MUST satisfy Document conformance and declare an Execution profile consistent with the SHACL constructs used.
SPL PDP (processor)A Policy Decision Point implementation that evaluates SPL policies against shpl:AccessRequest graphs.MUST satisfy Processor (PDP) conformance, Evaluation semantics, Node expression hooks (for hooks it implements), Holder binding, and Verifiable Credentials as applicable.
SPL test processorA test harness or continuous-integration runner that claims to verify SPL behavior.MUST satisfy the minima in Appendix D for any test it reports as “SPL v0.3 conformant”.

Policy Enforcement Points, Policy Administration Points, and Policy Information Points are not separate SPL conformance classes in v0.3; their obligations are architectural (see Conceptual alignment) and MAY be constrained by companion profiles.

Execution profiles

SPL-Core uses only SHACL 1.2 Core, Rules, and Node Expressions. SPL-Full additionally allows SHACL SPARQL Extensions.

Each SPL policy MUST declare exactly one shpl:profile with value shpl:Core or shpl:Full.

SPL-Core policies MUST NOT use SHACL constructs whose semantics depends on SPARQL Extensions.

Document conformance

A conformant SPL policy MUST validate against the SPL-SPL shapes graph in Appendix B plus any shpl:profileShapesGraph from declared domain profiles.

Each SPL policy MUST declare exactly one shpl:specVersion IRI.

Processor (PDP) conformance

A conformant PDP MUST: evaluate shpl:condition with SHACL 1.2; classify each shpl:Rule per Evaluation semantics; apply shpl:detectConflicts and the SPL policy’s shpl:conflictStrategy (default shpl:DenyOverrides); verify credentials before they influence a satisfied shpl:Permit or a satisfied shpl:Deny whose Rule condition depends on credential-derived triples (see Credential verification and holder binding); enforce holder binding per Holder binding; honor shpl:dryRun; materialize duties from SHACL rules; and return a Decision graph conforming to AccessRequest and Decision.

A conformant PDP SHOULD publish a shpl:CapabilityDescriptor (see Capability descriptor) at an IRI documented by the deployment. When a descriptor is published, it MUST validate against shpl:CapabilityDescriptorShape in Appendix B. How clients discover that IRI is specified by companion profiles, not by this core document.

When a PDP does not publish a capability descriptor, clients MUST assume it supports only the minimum SPL-Core feature set implied by this specification; every shpl:requiredCapability IRI in an SPL policy MUST be treated as an unsupported Capability unless otherwise proven out-of-band.

Vocabulary

This section normatively defines the shape and meaning of core shpl: terms. For motivation and scope, see Goals and background; for defined words such as PDP and PEP, see Terminology.

Namespace https://w3id.org/shacl-policy-language# (shpl:). Canonical Turtle: spl.ttl.

Policy and rules

shpl:Policy
Carries exactly one shpl:profile, exactly one shpl:specVersion, optional shpl:conformsToProfile, shpl:requiredCapability (each value names a required Capability), shpl:conflictStrategy, shpl:detectConflicts, and shpl:usesActionVocabulary.
shpl:Rule
The single rule class in core. A rule has an shpl:effect, target, action, optional SHACL condition, and optional attached duties through shpl:duty.
shpl:effect
Exactly one of shpl:Permit, shpl:Deny, or shpl:Obligate. Permit and Deny contribute access decisions. Obligate materializes obligations but does not grant access by itself.
shpl:target
One or more IRIs naming the governed resource (class membership or shpl:AssetCollection allowed). In [[ODRL]] terms the same notion is an asset; authors SHOULD ensure targets can be understood as odrl:Asset where interoperability with ODRL tools matters.
shpl:action
Exactly one action IRI, with ODRL recommended.
shpl:condition
Optional sh:NodeShape defining the Rule condition. If absent, condition evaluation is satisfied when target and action match.
shpl:duty
Optional authoring-time link from a rule, or from another duty, to a shpl:Duty specification.
shpl:usesActionVocabulary
Declares vocabulary graph IRIs whose action-closure triples are in scope for evaluation.

Duties and obligations

shpl:duty is the authoring-time link from a rule or duty to a shpl:Duty specification. shpl:obligation links a shpl:Decision to obligation nodes (a shpl:Obligation). This separation keeps duty specifications (policy) and obligations (decision output) distinct, as with obligations on an authorization decision in [[XACML30]]; see Conceptual alignment for obligations versus advice.

shpl:Duty is a Duty specification. Each duty MUST declare exactly one shpl:triggersOn value:

  • shpl:OnRuleApplied — materialize when the attached rule is satisfied and its effect realizes.
  • shpl:OnRuleViolated — materialize as breach-response output when the attached satisfied rule contributes shpl:Deny.
  • shpl:OnObligationFailed — materialize when a parent obligation reaches shpl:ObligationFailed.

shpl:timing is shpl:Precondition or shpl:Postcondition. The same property is used on a shpl:Duty and on each shpl:Obligation node in the Decision.

Obligation state on each shpl:Obligation node: shpl:ObligationPending, shpl:ObligationInProgress, shpl:ObligationSatisfied, shpl:ObligationFailed, shpl:ObligationCompensated. The PEP advances obligation states as it enforces side effects. A duty attached to another duty with trigger shpl:OnObligationFailed is evaluated when the parent obligation fails.

Core duties MUST emit obligation payload triples using SHACL sh:rule. Compact direct predicates and domain-specific obligation verbs are companion-profile terms, not core vocabulary.

AccessRequest and Decision

shpl:AccessRequest
Exactly one shpl:agent, shpl:target, and shpl:action; zero or more shpl:credential; optional shpl:dryRun and shpl:explanationLevel. If the client presents a Verifiable Presentation, the PDP MUST expand member credentials into shpl:credential before evaluation.
shpl:Decision
Exactly one shpl:outcome whose value is a shpl:DecisionOutcome individual (shpl:Permit, shpl:Deny, shpl:NotMatched, shpl:NotSatisfied, or shpl:Indeterminate); exactly one shpl:requestRef; zero or more shpl:reason; zero or more shpl:appliedRule; zero or more shpl:obligation nodes typed shpl:Obligation; optional shpl:explanation.
shpl:explanationLevel
shpl:NoExplanation, shpl:SummaryExplanation, or shpl:FullExplanation.
shpl:Core / shpl:Full
The only two shpl:ExecutionProfile individuals; values of shpl:profile on a shpl:Policy. Domain extension manifests use shpl:Profile, which is not a superclass of execution profiles.

Conflict strategies (core)

The v0.3 core defines two conflict strategies: shpl:DenyOverrides (default) and shpl:PermitOverrides. Each is an individual of shpl:ConflictStrategy. These are the deny-overrides and permit-overrides rule-combining / policy-combining strategies defined in [[XACML30]]; see Decisions and rule combining. An SPL policy MAY set shpl:detectConflicts true; when both Permit and Deny contributions are present, detection yields shpl:Indeterminate with shpl:reason including shpl:Conflict before the override strategy is applied. Strategy metadata such as determinism is descriptive and does not change evaluation outcomes.

When the shpl:Decision outcome is shpl:NotMatched or shpl:NotSatisfied, the PEP MUST treat the request as denied at the enforcement boundary.

Decision reasons

Decision reasons are SKOS concepts in the open shpl:DecisionReason concept scheme. Core concepts include shpl:Conflict, shpl:UnsupportedFeature, shpl:HolderBindingFailed, shpl:CredentialNotVerified, shpl:NodeExpressionFailed, shpl:CycleDetected, and shpl:RuleIndeterminate. Profiles MAY define additional reason concepts. shpl:reason carries coarse machine-readable tags; optional shpl:explanation carries structured detail such as shpl:missingFeature.

Enumerated value classes

The ontology declares closed wire enumerations with owl:oneOf: shpl:RuleEffect, shpl:DecisionOutcome, shpl:DutyTiming, shpl:ObligationTrigger, shpl:ObligationState, shpl:ConflictStrategy, shpl:ExplanationLevel, and shpl:ExecutionProfile. Decision reasons are intentionally open SKOS concepts. shpl:Permit and shpl:Deny are reused as both effects and outcomes because they denote the same access result; shpl:Obligate is only a rule effect and MUST NOT appear as shpl:outcome.

Term stability (spl.ttl)

Every normative SPL term carries shpl:stability with value shpl:Stable or shpl:Experimental. This is metadata for ontology consumers; it is not evaluated at decision time.

Capability descriptor

shpl:CapabilityDescriptor is an RDF description of what a PDP deployment implements: its Execution profile, supported specification versions, and optional shpl:processorVersion, shpl:supportedCapability, and shpl:supportedProfile (each shpl:supportedCapability value is a Capability IRI). Subclasses such as shpl:HookCapability, shpl:EvaluationCapability, shpl:ObligationCapability, shpl:ProfileCapability, and shpl:ActionReasoningCapability are for discovery and documentation; support for a given feature is still determined by the advertised Capability IRI.

Conditions (SHACL only)

The SHACL focus node for evaluating each rule's shpl:condition is the shpl:AccessRequest being decided (the same resource identified by shpl:requestRef in the eventual shpl:Decision). The Access request graph includes that node, attached credentials, and any deployment-supplied triples the PDP includes for evaluation.

Authors express requirements exclusively as sh:NodeShape values of shpl:condition (the Rule condition). A rule with no shpl:condition has nothing to validate in step 2 of Evaluation semantics; the PDP treats that as an immediately satisfied condition once target and action match.

The canonical pattern for "at least one credential satisfies S" is:

sh:property [
  sh:path shpl:credential ;
  sh:qualifiedValueShape [ a sh:NodeShape ; ... ] ;
  sh:qualifiedMinCount 1
] .

Node expression hooks

Standard hooks (see spl.ttl): shpl:currentDateTime, shpl:requestCount, shpl:agentLocation, shpl:credentialClaim, shpl:purposeOfUse, shpl:resourceOwner. These IRIs identify hook implementations; their hook-category typing (shpl:PureHook, shpl:RequestScopedHook, shpl:EnvironmentalHook, or shpl:ExternalHook) supports documentation, capability discovery, and deterministic profiles.

Hook evaluation contract

Hooks are invoked from SHACL node expressions (and related SHACL facilities) during Evaluation semantics step 2. A conformant PDP MUST treat hook resolution as part of SHACL validation of the Access request graph (plus policy and vocabulary triples the PDP loads for evaluation).

If evaluation reaches a hook IRI that the PDP does not implement, or that is not advertised in its shpl:CapabilityDescriptor when a descriptor is published, the PDP MUST classify the affected rule as Indeterminate (per-rule result) and SHOULD include shpl:reason shpl:UnsupportedFeature or shpl:NodeExpressionFailed on the shpl:Decision as appropriate.

If a hook implementation returns no value, an ill-typed literal, or values that violate the shape’s datatype or cardinality expectations, the PDP MUST treat condition evaluation as incomplete for that rule and MUST classify that rule as Indeterminate with shpl:reason including shpl:NodeExpressionFailed unless a companion profile defines a different deterministic error mapping.

If hook or SPARQL evaluation exceeds deployment limits (time, memory, result size, recursion depth), the PDP MUST NOT silently substitute a guessed value; it MUST yield shpl:Indeterminate with shpl:reason that includes shpl:UnsupportedFeature or shpl:NodeExpressionFailed (and MAY use shpl:explanation for detail).

For shpl:dryRun true, environmental and external hooks MUST NOT persist side effects outside the PDP process (for example network callbacks that mutate external state). They MAY still return values required for SHACL evaluation to complete. Companion profiles MAY define stronger dry-run semantics for obligations and enforcement.

shpl:PureHook and shpl:RequestScopedHook implementations SHOULD be deterministic given the same Access request graph and policy inputs. shpl:EnvironmentalHook and shpl:ExternalHook MAY vary over time; policies that depend on them SHOULD account for shpl:Indeterminate when values are unavailable.

Values for hooks that read deployment or environment state are typically obtained from a Policy Information Point (PIP) [[XACML30]] when they are not already present as literals or triples in the Access request graph. Such values contribute to environment (and, where relevant, resource or subject) attribute categories in the ABAC sense [[NIST-SP-800-162]]. Which hooks a deployment resolves via a PIP versus pre-materializing into the Access request graph is implementation-defined; the observable SHACL evaluation result MUST still match Evaluation semantics.

Actions

Normatively, v0.3 uses shpl:impliedAction (a transitive property, declared owl:inverseOf odrl:includedIn) for broader-than request actions. Applicability: a rule with action A matches request action R if R = A or A shpl:impliedAction+ R over the policy graph plus action vocabulary graphs declared by shpl:usesActionVocabulary and loaded by the PDP for evaluation.

Profiles MAY additionally publish the same hierarchy using skos:broader / skos:narrower on action IRIs; that does not replace the normative shpl:impliedAction rule unless a companion profile defines a deterministic mapping.

Verifiable Credentials

RDF representation

Credentials MUST appear as cred:VerifiableCredential nodes linked from the request via shpl:credential.

Holder binding

The PDP MUST ensure each cred:VerifiableCredential that is used to justify a shpl:Permit is bound to the request's shpl:agent before that credential's claims may satisfy a Rule condition.

Binding is satisfied when the holder subject identifier equals shpl:agent, where the holder subject identifier is, in order of availability: (1) the credential subject IRI if cred:credentialSubject is an IRI node; (2) the subject's cred:id if present on a blank-node subject; (3) another deployment-chosen stable identifier on the subject node that the PDP documents as equivalent to shpl:agent for binding purposes. If no holder subject identifier can be determined, the credential MUST NOT be used for Permit and the PDP SHOULD emit shpl:Deny or shpl:Indeterminate with shpl:reason including shpl:HolderBindingFailed.

Implementations MAY enforce binding by graph construction, by SHACL shapes on shpl:credential, or by an explicit pre-validation step; the mechanism is implementation-defined as long as the observable decision matches this clause.

Delegation and on-behalf-of flows are deferred to companion profiles.

Evaluation semantics

These steps define how a conformant PDP arrives at a shpl:Decision from a shpl:Policy and an Access request graph. They realize the policy evaluation role described under Goals and background and Conceptual alignment. The ordered list below is the authoritative evaluation algorithm; following subsections refine edge cases (credentials, outcomes, indeterminacy, obligations) without replacing those steps.

Each rule is classified independently. First determine applicability (whether the request's resource, action, and target identity match the rule so the rule is in scope for evaluation; the same notion as whether a policy or rule applies in [[XACML30]]). Then, if present, evaluate the Rule condition (SHACL constraints over the attribute context, in the ABAC sense [[NIST-SP-800-162]]). Together they yield a per-rule result:

  • Permit — effect shpl:Permit, target/action matched, and condition satisfied.
  • Deny — effect shpl:Deny, target/action matched, and condition satisfied.
  • Obligate — effect shpl:Obligate, target/action matched, and condition satisfied; this may materialize obligations but does not grant access.
  • NotMatched — target/action did not match.
  • NotSatisfied — target/action matched, but the SHACL condition was violated.
  • Indeterminate — condition or hook evaluation could not complete, or a structural error was detected.
  1. Applicability — A rule is applicable when the request's shpl:target and shpl:action match the rule's target set and action under resource identity, class membership, collection membership, and action-closure rules in Actions. If not applicable, the per-rule result is NotMatched.
  2. Condition evaluation — If applicable and shpl:condition is present, run SHACL validation of that sh:NodeShape over the Access request graph. If the shape is violated, the per-rule result is NotSatisfied. If validation cannot complete, the result is Indeterminate with an appropriate reason. If shpl:condition is absent, the condition is satisfied.
  3. Conflict detection — If shpl:detectConflicts true and, among rules classified as satisfied Permit or satisfied Deny, both kinds are present, the PDP yields shpl:Indeterminate with reason shpl:Conflict (before applying the override strategy in step 4).
  4. Conflict strategy — Otherwise, apply the SPL policy's shpl:conflictStrategy to satisfied Permit and satisfied Deny rules (after step 3). Under shpl:DenyOverrides, any satisfied Deny yields shpl:Deny; else any satisfied Permit yields shpl:Permit. Under shpl:PermitOverrides, any satisfied Permit yields shpl:Permit; else any satisfied Deny yields shpl:Deny. If no rule was applicable on target and action, set shpl:outcome to shpl:NotMatched. If at least one rule was applicable but neither a satisfied Permit nor a satisfied Deny is produced for the strategy, set shpl:outcome to shpl:NotSatisfied or shpl:Indeterminate as specified in Access outcome from Permit, Deny, and Obligate and Indeterminate propagation.
  5. Obligation materialization — Materialize duties attached through shpl:duty using their shpl:triggersOn value and their SHACL sh:rule payloads. Duties triggered by shpl:OnRuleApplied materialize when the attached rule's effect realizes. Duties triggered by shpl:OnRuleViolated materialize from satisfied Deny rules. Duties triggered by shpl:OnObligationFailed materialize when the parent obligation fails.

Step 5 runs after steps 1–4 establish the consolidated shpl:outcome. When step 3 yields shpl:Indeterminate with shpl:reason including shpl:Conflict, step 4 does not select shpl:Permit or shpl:Deny, and the PDP MUST NOT materialize duties whose shpl:triggersOn is shpl:OnRuleApplied or shpl:OnRuleViolated from rules classified in steps 1–2 as satisfied Permit or satisfied Deny among the set step 3 tested (the conflicting contributors). The PDP MUST still materialize obligations from rules classified as satisfied Obligate in steps 1–2. Nested duties and shpl:OnObligationFailed behavior otherwise follow Obligation materialization (multiple and nested duties).

Credential verification and holder binding

Before classifying any rule as a satisfied Permit whose Rule condition depends on triples originating from a shpl:credential value, the PDP MUST complete any credential verification steps the deployment associates with that credential (for example cryptographic proof verification, schema checks, and revocation or status checks where applicable). If verification fails for a credential whose claims were needed for that Permit, the PDP MUST NOT classify that rule as satisfied Permit; it SHOULD treat the rule as NotSatisfied for access purposes unless another outcome is required to surface the failure, in which case it MUST use Indeterminate with shpl:reason including shpl:CredentialNotVerified.

When a Deny rule's Rule condition depends on triples originating from a shpl:credential value, the PDP MUST complete the same credential verification steps the deployment associates with that credential (see Permit credential verification) before classifying that rule as satisfied Deny. If verification fails, the PDP MUST NOT classify that rule as satisfied Deny; it SHOULD treat the rule as NotSatisfied for access purposes or, when the deployment must surface verification failure distinctly, as Indeterminate with shpl:reason including shpl:CredentialNotVerified.

Deployments that require pairwise-comparable PDP outputs on verification failure SHOULD fix the choice between NotSatisfied and Indeterminate in deployment policy or in a companion profile.

Holder binding applies as specified independently of verification: even a verified credential MUST NOT justify Permit unless bound to shpl:agent.

Access outcome from Permit, Deny, and Obligate

Steps 3 and 4 consider only rules whose per-rule classification is satisfied Permit or satisfied Deny (that is, applicable with a satisfied condition, including the vacuous case where no shpl:condition is present). Obligate rules do not set shpl:outcome to shpl:Permit or shpl:Deny; they never supply the Permit or Deny contributions used by conflict detection or override strategies.

If at least one rule is applicable on target and action but every such rule is either Obligate, NotSatisfied, or Indeterminate, and no satisfied Permit or Deny results, the consolidated shpl:outcome after step 4 is shpl:NotSatisfied unless step 2 already forced shpl:Indeterminate for a rule that blocks completion.

If no rule is applicable (all NotMatched on target and action), the consolidated outcome is shpl:NotMatched.

Indeterminate propagation

Let deciding mean: a satisfied Deny that fixes shpl:outcome to shpl:Deny under shpl:DenyOverrides, or a satisfied Permit that fixes shpl:outcome to shpl:Permit under shpl:PermitOverrides, per step 4.

If any applicable rule has per-rule result Indeterminate from step 2 (including from hooks per Node expression hooks), and that indeterminacy prevents establishing whether the rule would contribute a satisfied Permit, Deny, or Obligate, the PDP MUST set shpl:outcome to shpl:Indeterminate unless a companion profile defines a safe fallback that remains conformant with this clause.

If some rules are Indeterminate but a satisfied Deny is already deciding under shpl:DenyOverrides, the outcome remains shpl:Deny and shpl:reason MAY also include reasons associated with indeterminate rules. Symmetrically under shpl:PermitOverrides with a deciding Permit. If neither override side can be resolved because of indeterminate inputs, the outcome is shpl:Indeterminate.

Obligation materialization (multiple and nested duties)

The PDP MUST emit distinct shpl:Obligation nodes for each materialized duty instance required by the policy and evaluation result, including multiple duties attached to the same rule. Reusing one obligation node for several independent duties is a violation of this intent and of typical SPL-SPL validation expectations.

For shpl:duty values nested on a shpl:Duty with shpl:OnObligationFailed, initial access-time evaluation emits obligations for shpl:OnRuleApplied and shpl:OnRuleViolated as above. Materializing nested OnObligationFailed duties against a parent obligation that is not yet failed is out of scope for the initial shpl:Decision unless the Access request graph already includes shpl:Obligation nodes in shpl:ObligationFailed state (for example a re-evaluation call). Companion profiles MAY define how PEP-reported failures trigger a subsequent PDP invocation.

Profiles and extensions

shpl:ExecutionProfile (shpl:Core, shpl:Full) is declared on the SPL policy via shpl:profile as its Execution profile. Domain profiles are shpl:Profile resources referenced by shpl:conformsToProfile, carrying shpl:declaresCapability, shpl:appliesToExecutionProfile, and shpl:profileShapesGraph as needed. A shpl:CapabilityDescriptor summarizes PDP capabilities.

Versioning

SPL policies pin shpl:specVersion to an IRI of the form https://w3id.org/shacl-policy-language/spec/vmajor.minor.patch. In spl.ttl, shpl:specVersion and shpl:conformsToProfile are asserted as rdfs:subPropertyOf dct:conformsTo, so generic Dublin Core queries also see spec and domain profile pins on an SPL policy.

Security considerations

Untrusted graphs

SPL policies, Access request graph inputs, and referenced vocabulary graphs MUST be obtained from trust anchors appropriate to the deployment (for example signed artifacts, authenticated endpoints, or local sealed stores). A PDP that evaluates attacker-controlled Turtle can be made to load arbitrary IRIs, shapes, and action-closure triples, which becomes remote code or data exfiltration when combined with SPARQL extensions or external hooks.

SHACL and SPARQL surfaces

SPL-Full allows SHACL constructs whose semantics depends on [[SHACL12-SPARQL]]. Treat SPARQL execution in policy evaluation as a privilege boundary: cap time and memory, forbid or sandbox SERVICE unless explicitly allowed, and log policy identifiers that contributed to queries. SPL-Core reduces this surface but does not remove denial-of-resource risks from pathological shapes; deployments SHOULD still bound validation work.

Hooks and external calls

shpl:EnvironmentalHook and shpl:ExternalHook are natural attachment points for network or system calls. Implementations MUST authenticate and authorize those calls, apply timeouts, and avoid forwarding unsanitized request attributes into queries or logs. See Node expression hooks for normative failure behavior when hooks are missing or exceed limits.

Decisions at trust boundaries

Decision graphs SHOULD be integrity-protected when consumed outside the PDP process (for example signatures, TLS within a trust domain, or append-only audit logs). Without that, an attacker who can inject or modify in-flight RDF can change shpl:outcome or strip obligations. Binding decisions to the originating request (shpl:requestRef) and policy version (shpl:specVersion on the evaluated policy) helps downstream systems detect replay or stale evaluation.

Credential and policy misuse

Even with Holder binding and Credential verification, deployments MUST still enforce issuer trust, audience or scope constraints, and revocation outside SPL’s core vocabulary. Policies that emit rich shpl:explanation graphs can become a channel for leaking sensitive attribute data; default to shpl:NoExplanation for Internet-facing PDPs unless a risk assessment says otherwise (Privacy considerations).

Privacy considerations

Data minimization

Clients SHOULD attach only the credentials and claims strictly needed for the decision. PDPs SHOULD evaluate conditions against the smallest materialized graph practical and SHOULD NOT retain request or decision graphs longer than policy, law, or audit schedules require.

Correlation and linkability

Stable shpl:agent IRIs, credential identifiers, and repeated shpl:requestRef patterns enable correlation across requests. Deployments that need anonymity or unlinkability require controls outside SPL core (for example pairwise identifiers, selective disclosure, or minimal proofs) via Companion profiles.

Explanations and logging

shpl:SummaryExplanation and shpl:FullExplanation can embed attribute values, paths, or subgraphs that are sensitive in regulated contexts. PEPs and log sinks SHOULD treat explanation payloads as personal or security-sensitive data and apply redaction, access control, and retention limits. shpl:Indeterminate responses SHOULD carry only the minimum detail needed for operators to act on shpl:reason.

Hook-derived attributes

Hooks that read location, purpose of use, or environment can reveal sensitive context when echoed into decisions or obligations. PDPs SHOULD document which hook values appear in Decision graph outputs and offer configuration to suppress them.

Examples

The fragments below are informative. A complete SPL policy still needs shpl:profile, shpl:specVersion, and other triples on the policy resource as in SPL in five minutes.

Example 1: Permit read if age ≥ 18

Same content as SPL in five minutes: a shpl:Permit on ex:AdultOnly and odrl:read with a shpl:condition that requires at least one credential whose subject has ex:age at least 18, plus a postcondition shpl:duty that adds an audit-sink triple when the rule applies.

Example 2: Deny if under 13

A shpl:Deny on the same target and action, but the condition matches when some credential subject has ex:age strictly less than 13. When that shape is satisfied, the rule contributes Deny (child-safety style guard). If no credential matches, this rule’s condition is not satisfied, so this fragment alone does not force Deny—final outcome depends on other rules and combination semantics in Evaluation semantics.

ex:Under13 a shpl:Rule ;
  shpl:effect shpl:Deny ;
  shpl:target ex:AdultOnly ;
  shpl:action odrl:read ;
  shpl:condition [
    a sh:NodeShape ;
    sh:property [
      sh:path shpl:credential ;
      sh:qualifiedValueShape [
        a sh:NodeShape ;
        sh:property [
          sh:path cred:credentialSubject ;
          sh:node [ sh:property [ sh:path ex:age ; sh:maxExclusive 13 ] ]
        ]
      ] ;
      sh:qualifiedMinCount 1
    ]
  ] .

Example 3: Two credentials (disjoint qualified shapes)

Intent: allow read on ex:ResearchDb only when the request lists two distinct shpl:credential nodes—one proving adulthood and one proving ex:clearance is ex:Secret. Without disjointness, one credential satisfying both inner shapes could satisfy both counts.

Two sh:property constraints on shpl:credential, each with sh:qualifiedMinCount 1, do not by themselves require two different credentials: one credential that satisfies both inner shapes would satisfy both. Set sh:qualifiedValueShapesDisjoint true on the surrounding sh:NodeShape so that distinct values are required for each qualified shape at the same path (each value is validated as its own focus node in SHACL).

ex:TwoVC a shpl:Rule ;
  shpl:effect shpl:Permit ;
  shpl:target ex:ResearchDb ;
  shpl:action odrl:read ;
  shpl:condition [
    a sh:NodeShape ;
    sh:qualifiedValueShapesDisjoint true ;
    sh:property [
      sh:path shpl:credential ;
      sh:qualifiedValueShape [
        a sh:NodeShape ;
        sh:property [ sh:path cred:credentialSubject ;
          sh:node [ sh:property [ sh:path ex:age ; sh:minInclusive 18 ] ] ]
      ] ;
      sh:qualifiedMinCount 1
    ] ;
    sh:property [
      sh:path shpl:credential ;
      sh:qualifiedValueShape [
        a sh:NodeShape ;
        sh:property [ sh:path cred:credentialSubject ;
          sh:node [ sh:property [ sh:path ex:clearance ; sh:hasValue ex:Secret ] ] ]
      ] ;
      sh:qualifiedMinCount 1
    ]
  ] .

Example 4: Permit + postcondition audit duty

ex:LogDuty defines a shpl:Duty that, when materialized, uses sh:TripleRule to add an ex:auditSink triple (log endpoint) on the obligation. ex:ReadAndLog is a shpl:Permit for odrl:read on ex:Document with no shpl:condition, so it applies on every matching read; it references ex:LogDuty so a successful permit also yields a postcondition obligation for auditing.

ex:LogDuty a shpl:Duty ;
  shpl:triggersOn shpl:OnRuleApplied ;
  shpl:timing shpl:Postcondition ;
  sh:rule [
    a sh:TripleRule ;
    sh:subject sh:this ;
    sh:predicate ex:auditSink ;
    sh:object <https://example.com/audit>
  ] .

ex:ReadAndLog a shpl:Rule ;
  shpl:effect shpl:Permit ;
  shpl:target ex:Document ;
  shpl:action odrl:read ;
  shpl:duty ex:LogDuty .

No shpl:condition means the rule applies whenever target and action match.

Example 5: Deny with breach-response duty

ex:NoExternalRead denies odrl:read on ex:InternalDoc with no condition (any applicable read is prohibited). ex:NotifyDuty is triggered with shpl:OnRuleViolated: when this Deny rule is satisfied and contributes, the duty’s rule emits a ex:notify triple (for example alerting security)—a breach-response pattern on top of Deny.

ex:NotifyDuty a shpl:Duty ;
  shpl:triggersOn shpl:OnRuleViolated ;
  shpl:timing shpl:Postcondition ;
  sh:rule [
    a sh:TripleRule ;
    sh:subject sh:this ;
    sh:predicate ex:notify ;
    sh:object <mailto:security@example.com>
  ] .

ex:NoExternalRead a shpl:Rule ;
  shpl:effect shpl:Deny ;
  shpl:target ex:InternalDoc ;
  shpl:action odrl:read ;
  shpl:duty ex:NotifyDuty .

Example 6: Decision graph

Illustrates a shpl:Decision after a permit: shpl:outcome, shpl:appliedRule pointing at the winning rule, an shpl:obligation block in shpl:ObligationPending with payload triples for the PEP, and shpl:requestRef linking to the shpl:AccessRequest.

ex:Dec a shpl:Decision ;
  shpl:outcome shpl:Permit ;
  shpl:appliedRule ex:AdultRead ;
  shpl:obligation [
    a shpl:Obligation ;
    shpl:fromRule ex:AdultRead ;
    shpl:timing shpl:Postcondition ;
    shpl:state shpl:ObligationPending ;
    ex:auditSink <https://example.com/audit>
  ] ;
  shpl:requestRef ex:Req1 .

Example 6b: Deny with reason

A compact Deny decision: shpl:outcome shpl:Deny, a machine-readable shpl:reason (shpl:CredentialNotVerified), shpl:appliedRule naming the contributing rule, and shpl:requestRef for traceability—typical when the PDP denies without materializing obligations.

ex:DecDeny a shpl:Decision ;
  shpl:outcome shpl:Deny ;
  shpl:reason shpl:CredentialNotVerified ;
  shpl:appliedRule ex:Under13 ;
  shpl:requestRef ex:Req1 .

Constraints → SHACL (pattern)

ODRL odrl:constraint left/right/operator triples do not appear in SPL v0.3 core. Map each constraint to a sh:property (or sh:sparql in SPL-Full) on the Access request graph. Typical patterns:

ODRL patternSHACL sketch

Not mapped in core

ODRL defines policy negotiation artifacts such as odrl:Offer, odrl:Request, and odrl:Agreement, optional human-oriented metadata, and patterns for composing or comparing policies outside a single PDP invocation [[ODRL]]. SPL v0.3 core deliberately does not fix lifecycle protocols, merge semantics, or legal prose: it specifies only how a shpl:Policy is validated and evaluated against an shpl:AccessRequest to produce a shpl:Decision (the same separation as between transport and the XACML reference architecture in Reference architecture (XACML)).

A companion profile (see Companion profiles, https://w3id.org/shacl-policy-language/profile/negotiation) or another specification MAY define how negotiation state is represented in RDF, how an odrl:Agreement is projected to or linked with a shpl:Policy, or how multiple policy sources are combined before evaluation. Until such a profile exists, implementations that need ODRL negotiation SHOULD use ODRL and related specs for interchange and treat SPL as the evaluation and decision layer.

Appendix D: Conformance test suite

This appendix states the minimum expectations for a conformant test runner that claims SPL v0.3 compatibility. A richer protocol (RDF canonicalization, deterministic hook bindings, interchange of implementation-specific triples) remains reserved to https://w3id.org/shacl-policy-language/profile/conformance-tests.

  • Policy validation — Before evaluation, each input SPL policy graph under test MUST be validated against the normative SPL-SPL shapes graph in Appendix B plus any shpl:profileShapesGraph declared by domain profiles referenced in the test fixture.
  • Request validation — Each shpl:AccessRequest fixture SHOULD validate against SPL-SPL where shapes exist for requests.
  • Evaluation — The runner MUST apply Evaluation semantics (including hooks and Holder binding) using a PDP configuration documented with the test (execution profile, supported capabilities, and any hook bindings).
  • Decision check — For tests that ship an expected Turtle graph under tests/expected/, the runner MUST compare the produced Decision graph using full RDF isomorphism after a documented normalization step (for example stable blank-node skolemization or canonical RDF forms agreed for that test corpus). If normalization is not yet fixed by the companion profile, the test manifest MUST state the normalization algorithm used.
  • Negative tests — Policies or requests that are intentionally invalid MUST fail SPL-SPL validation before any evaluation step is applied.

Tests under tests/ use manifests in tests/manifests/; this repository provides fixtures that illustrate the above expectations.

Appendix F: Document conventions

  • Requirements — Normative requirements use BCP 14 keywords in capitals and are often marked with class="req" and a stable id for cross-referencing.
  • Examples — Turtle, JSON-LD, and tabular examples marked informative are not normative unless they duplicate a normative clause verbatim.
  • Algorithms — The ordered list in Evaluation semantics is the normative evaluation algorithm; subsections refine outcomes without replacing those steps.
